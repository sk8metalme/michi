# Document Review Rules

## 目的

ドキュメント品質を自動的にチェックし、一貫性のある高品質なドキュメント作成を支援する。

## 文量ガイドライン

| ドキュメント種別 | 目安行数 | 必須セクション |
|------------------|----------|----------------|
| README | 30-50行 | 概要, 使い方, ライセンス |
| API仕様 | 機能あたり10-20行 | エンドポイント, パラメータ, レスポンス |
| 設計書 | 50-100行 | 目的, 構成, 判断理由 |
| 変更履歴 | 項目あたり1-3行 | バージョン, 変更内容 |

## 禁止パターン

以下のパターンを検出した場合、修正を提案する：

1. **冗長表現**
   - 「〜することができます」 → 「〜できます」に修正
   - 「〜という」「〜のような」の過剰な使用 → 削除

2. **過剰な背景説明**
   - 導入部での不必要な前置き
   - 自明な内容の長い説明

3. **自明なコードコメント転記**
   - コードそのままの説明
   - 変数名や関数名を繰り返すだけの説明

4. **重複記載**
   - 同じ情報が複数セクションに記載されている
   - 情報は1箇所にのみ記載すべき

## 必須セクション検証

### README
- [ ] 概要
- [ ] 使い方（インストール・実行方法）
- [ ] ライセンス

### API仕様
- [ ] エンドポイント
- [ ] パラメータ
- [ ] レスポンス

### 設計書
- [ ] 目的
- [ ] 構成
- [ ] 判断理由

### 変更履歴
- [ ] バージョン
- [ ] 変更内容

## レビュー結果出力形式

```markdown
## ドキュメントレビュー結果

### 文量チェック
✅ OK (45行) / ⚠️ 超過 (65行、目標50行以内) / ❌ 不足 (20行、最低30行必要)

### 禁止パターンチェック
✅ 問題なし

または

⚠️ 冗長表現: 3箇所
  - L12: 「することができます」 → 「できます」
  - L28: 「という」 → 削除を検討
  - L35: 「することができます」 → 「できます」

### 必須セクションチェック
✅ すべて揃っています

または

❌ 必須セクション不足: ライセンス

### 総合評価
✅ 合格 / ⚠️ 改善推奨 / ❌ 不合格
```

## レビュー実行タイミング

- Markdownファイル作成時
- Markdownファイル編集完了時
- 手動レビューコマンド実行時（`/doc:review`）

## 適用除外

以下のファイルはレビュー対象外：
- `CHANGELOG.md` - 自動生成される場合
- `node_modules/**/*.md` - 依存パッケージのドキュメント
- `.git/**/*.md` - Git管理ファイル
- `vendor/**/*.md` - サードパーティライブラリ
